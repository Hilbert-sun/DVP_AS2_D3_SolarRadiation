## Section 2: Design Rationale Replacement / Expansion

The design rationale was guided by Munzner's what-why-how framework. The primary data abstractions are temporal daily observations and derived annual, monthly, seasonal, and extreme-event summaries. The main user tasks are to identify trends, compare seasonal rhythms, relate sunshine duration to solar radiation, and inspect high-radiation events. These tasks led to line charts for ordered temporal comparison, a multivariate scatterplot for relationship analysis, and summary cards plus a bar chart for extreme-day characterisation. This mapping avoids choosing chart types only by convention; each visual form is tied to a specific analytical task and audience need.

The interface uses a guided narrative structure rather than a neutral dashboard because the intended audience consists of urban planners and policy advisors who need interpretable evidence for decision-making, not unrestricted statistical exploration. The five story steps follow a deliberate sequence from context to implication: long-term variability, seasonal rhythm, daily driver, extreme events, and planning takeaway. This sequence manages cognitive load by introducing simple temporal patterns before asking the reader to interpret the denser multivariate scatterplot.

Several design principles from the unit informed the final layout. Tufte's principle of maximising meaningful data ink influenced the restrained visual style, limited decoration, and emphasis on labelled axes, legends, and direct annotations. Gestalt principles of proximity and similarity shaped the placement of controls, story navigation, and the insight panel: related controls are grouped together, repeated story buttons share a consistent visual treatment, and the active step is highlighted so that readers can maintain orientation. The visual hierarchy places the main chart at the centre, with narrative interpretation in a separate side panel, reducing competition between explanatory text and encoded data.

Colour and visual variables were selected conservatively to support interpretation rather than decoration. In the annual and seasonal views, hue separates variables while the common standardised scale supports comparison across different units. In the scatterplot, horizontal position encodes sunshine duration and vertical position encodes global solar radiation because position on a common scale is more accurate than area or colour for quantitative comparison. Rainfall is encoded by point size and humidity by colour to add weather context without requiring additional panels. These encodings are not equally precise, but they match the narrative hierarchy: the SUN-GSR relationship is primary, while rainfall and humidity are contextual.

The final implementation also strengthens the connection between summary evidence and individual records. In the extreme-day view, the top-10 table provides concrete daily examples. Clicking a date links back to the scatterplot and highlights the same day within the full daily context. This interaction supports Shneiderman's overview-then-details principle: readers first see the aggregate distribution of extreme days, then inspect a specific event, then compare it against the broader relationship between sunshine duration and solar radiation.

## Section 3.1: Technical Implementation Expansion

The implementation is modular to improve robustness and marker readability. `main.js` controls application state and re-renders views when filters change. `dataProcessing.js` handles CSV parsing, type conversion, season derivation, filtering, aggregation, and extreme-day identification. Each visualisation view is separated into its own module, reducing coupling between the annual trend, seasonal rhythm, scatterplot, and extreme-day components. This structure makes the implementation easier to inspect than a single large script and demonstrates deliberate data and interaction management.

D3.js is included locally in `js/lib/d3.v7.min.js` so the submitted project does not depend on a remote CDN during marking. The app reads all data from submitted CSV files and does not require server-side code, a database, or remote APIs. A local static server is only required because browser security rules restrict loading CSV files directly from the filesystem.

The main technical challenge was maintaining a consistent narrative state across multiple views. The year range, season filter, extreme-day highlight, and selected extreme date all interact with the same filtered dataset. The implementation therefore marks extreme days after filtering and passes the same state object to each renderer. The linked interaction between the Step 4 table and Step 3 scatterplot required the selected date to persist across views, allowing a specific extreme event to be traced from a summary table back to the multivariate daily relationship.

## Section 3.2: Interaction Rationale Expansion

The interaction design is intentionally constrained. The story buttons provide navigation through a fixed narrative sequence, while the filters support controlled comparison without turning the project into a general-purpose dashboard. This is important for the defined audience because planners need clear takeaways supported by evidence, not an interface that requires them to discover the story from scratch.

The extreme-day toggle and linked top-10 table add a second layer of interaction. When the toggle is active, extreme days are highlighted while non-extreme days remain visible as context. This avoids a common problem in filtered dashboards where removing the background data makes exceptional events look less meaningful. The linked date selection then lets the reader inspect one high-radiation event in relation to sunshine, rainfall, humidity, and the broader point cloud.

## Section 4: Reflection Expansion

One important design trade-off was between analytical completeness and narrative clarity. A more technical interface could have included regression models, seasonal correlation matrices, or additional meteorological variables. However, this would have increased cognitive load and moved the project away from the needs of planning and policy audiences. The final design therefore prioritises a coherent explanation of the strongest exploratory findings rather than presenting every possible analysis.

If extended further, the most valuable improvement would be stronger linked brushing across all views. For example, selecting a seasonal bar or an extreme-day record could highlight the same subset across the annual trend, monthly rhythm, scatterplot, and extreme-day table. This would make the application closer to a coordinated multiple-view system. A second improvement would be to incorporate local urban-form variables such as roof geometry, shading, or electricity demand so that the visualisation could move from climate-context evidence toward site-specific planning support.
